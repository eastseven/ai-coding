# AGENTS.md

## 1. Communication

- 默认使用中文回复。
- 回复要直接、结构化，优先给结论、依据、操作步骤。
- 不确定时明确说明不确定点，不要编造。
- 分析现有代码时，必须基于仓库中的真实代码，不要凭经验推断。

---

## 2. Documentation Policy

默认情况下，不要创建新的说明文档或文档文件。

不要自动生成或修改以下文件：

- `README.md`
- 设计文档
- 架构说明
- 使用说明
- 接口说明文档
- 部署说明文档
- 总结性文档

只有在用户明确要求以下操作时，才允许创建或修改说明文档：

- “编写文档”
- “生成 README”
- “写说明文档”
- “整理成文档”
- “输出成 md 文件”
- “创建设计文档”
- “更新架构说明”

### 2.1 状态文件例外

以下文件属于任务状态文件，不属于说明文档。

在满足本文件规则时，可以自动创建或修改：

- `tasks/todo.md`
- `tasks/archive/YYYY-MM.md`
- `tasks/lessons.md`

不要因为 Documentation Policy 而跳过这些状态文件。

---

## 3. Repository Behavior

修改代码前，先快速阅读相关代码，理解：

- 现有目录结构
- 命名风格
- 分层方式
- 依赖关系
- 已有实现习惯

代码修改原则：

- 保持现有项目风格。
- 修改范围最小化。
- 只处理当前任务要求的内容。
- 不做无关重构。
- 不顺手格式化无关文件。
- 不删除或覆盖用户已有改动。
- 不引入不必要的新依赖。
- 不改变已有公共接口行为，除非任务明确要求。

涉及数据库时：

- 任何表结构变更都必须提供 SQL 变更记录。
- SQL 需要考虑兼容性、默认值、历史数据影响。
- 字段命名避免使用数据库关键字，例如 `desc`、`index`、`order` 等。
- 如果存在升级风险，必须明确说明。

---

## 4. Code Style

- 代码应清晰、可维护。
- 保持项目现有命名、包结构、异常处理方式和返回格式。
- 新增函数、复杂分支、业务规则、状态流转、权限判断、边界条件必须写中文注释。
- 不要为简单 getter/setter、显而易见的赋值、框架样板代码添加噪音注释。
- 错误处理必须明确，不要吞异常。
- 参数校验、空值判断、权限判断、状态判断不能省略。
- 不要用临时 hack 掩盖根因。
- 优先找根因，再做修复。

---

## 5. Planning Policy

根据任务复杂度选择不同流程。

| 类型 | 判断标准 | Codex 行为 |
|---|---|---|
| 简单任务 | 单文件修改、小 bug、字段调整、样式调整、配置调整 | 写入极简 `tasks/todo.md` 后直接处理，完成后验证并清理 todo |
| 中等任务 | 涉及 2-5 个文件，包含接口、权限、状态流、数据库、前后端联动 | 写入 `tasks/todo.md`，给出简短计划，再实现 |
| 复杂任务 | 跨模块、跨系统、核心模型变化、权限体系变化、数据结构变化大、存在架构分歧 | 写入 `tasks/todo.md`，进入计划模式，等待用户确认后再实现 |

复杂任务计划必须包含：

1. 现状分析
2. 代码依据
3. 修改范围
4. 风险点
5. 验证方式

复杂任务在用户确认前，不要修改业务代码。

如果执行中发现计划不适用：

- 立即停止当前实现；
- 说明发现的偏差；
- 调整计划；
- 必要时等待用户确认。

## 6. Code Research Rules

分析现有代码时，必须基于真实代码。

任何关于现有实现的结论，都必须给出依据：

- 文件路径
- 类名 / 函数名 / 方法名
- 关键代码行为说明

不要使用以下无依据表达：

- “通常来说”
- “应该是”
- “看起来可能”
- “一般项目会这样”
- “我猜测”

如果没有找到依据，明确说明：

> 未在当前仓库中找到相关实现依据。

如果需要搜索代码：

- 优先搜索精确关键字；
- 再搜索相关类名、接口名、表名、字段名；
- 最后再根据调用链分析。

## 7. State Files Protocol

本节是硬性规则，用于解决 `tasks/todo.md` 和 `tasks/lessons.md` 经常未写的问题。

### 7.1 总原则

对任何会修改仓库文件的任务，必须使用 `tasks/todo.md`。

不要把 `tasks/todo.md` 当作长期历史记录。

`tasks/todo.md` 只保存当前任务状态。

任务完成后，必须：

1. 将结果归档到 `tasks/archive/YYYY-MM.md`；
2. 清空 `tasks/todo.md`，恢复为“当前没有进行中的任务”；
3. 检查是否需要写入 `tasks/lessons.md`；
4. 在最终回复中说明状态文件处理结果。

### 7.2 什么时候必须写 `tasks/todo.md`

以下情况必须创建或更新 `tasks/todo.md`：

- 需要修改任何代码文件；
- 需要修改配置文件；
- 需要修改 SQL、脚本、测试文件；
- 需要执行多步骤排查；
- 需要运行测试或构建验证；
- 用户要求“修复”“实现”“改造”“优化”“重构”“接入”“联调”；
- 任务可能跨多轮对话继续。

即使是简单代码修改，也要写入极简 todo。

### 7.3 什么时候可以不写 `tasks/todo.md`

只有以下情况可以不写：

- 用户只是咨询问题，不要求修改仓库；
- 用户只是要求解释代码；
- 用户只是要求阅读或总结已有内容；
- 用户明确要求“不要写任务文件”；
- 当前环境无法访问仓库文件系统。

如果不写 `tasks/todo.md`，最终回复中必须说明原因。

### 7.4 `tasks/todo.md` 写入方式

开始任务时，不要追加到旧内容后面。

必须用当前任务覆盖 `tasks/todo.md` 的主体内容，避免无限增长。

推荐格式：

```md
# TODO

## Current Task

目标：一句话描述当前任务。

## Plan

- [ ] 分析现有实现
- [ ] 定位影响范围
- [ ] 修改相关代码
- [ ] 补充或调整测试
- [ ] 运行验证
- [ ] 总结结果

## Related Files

- `path/to/file1`
- `path/to/file2`

## Risks

- 风险 1
- 风险 2

## Verification

- `命令 1`
- `命令 2`
```

简单任务可以使用极简格式：

```md
# TODO

## Current Task

目标：一句话描述当前任务。

## Plan

- [ ] 修改
- [ ] 验证
- [ ] 总结

## Verification

- 待执行
```

### 7.5 `tasks/todo.md` 更新时机

必须在以下节点更新 `tasks/todo.md`：

1. 开始修改代码前；
2. 完成关键步骤后；
3. 验证完成后；
4. 最终回复前。

最终回复前，如果任务已完成，`tasks/todo.md` 必须恢复为：

```md
# TODO

当前没有进行中的任务。
```

如果任务未完成，必须保留未完成项，并在最终回复中说明未完成原因。

### 7.6 归档规则

不要维护单个无限增长的：

```text
tasks/archive.md
```

归档必须按月份拆分：

```text
tasks/archive/YYYY-MM.md
```

例如：

```text
tasks/archive/2026-04.md
```

任务完成后，将简短结果追加到当月归档文件。

归档只保留结果，不保留过程流水。

推荐格式：

```md
## YYYY-MM-DD 任务标题

### Summary

- 简短说明完成了什么。

### Changed Files

- `path/to/file1`
- `path/to/file2`

### Verification

- 已运行：`命令`
- 结果：通过 / 失败 / 未运行及原因

### Notes

- 风险或后续建议。
```

### 7.7 `tasks/lessons.md` 强制规则

`tasks/lessons.md` 用于记录长期有效、可复用的规则。

以下情况必须写入或更新 `tasks/lessons.md`：

- 用户纠正了 Codex 的行为；
- 用户表达了长期偏好，例如“以后都这样”“下次不要”“默认要”“必须先做”；
- 本次任务暴露出可复用的项目规则；
- 因为错误假设导致返工；
- 验证失败暴露出后续必须避免的问题；
- 发现项目中特定框架、目录、命名、权限、数据库、测试方式的长期约束。

以下情况不要写入 `tasks/lessons.md`：

- 普通任务流水；
- 一次性 bug 描述；
- 临时日志；
- 无法复用的历史细节；
- 只是完成了某个任务。

### 7.8 `tasks/lessons.md` 写入格式

如果 `tasks/lessons.md` 不存在，且本次触发了 lessons 规则，必须创建。

推荐格式：

```md
# Lessons

## General

- 规则 1。
- 规则 2。

## Project Specific

- 项目特定规则 1。
- 项目特定规则 2。
```

新增 lesson 必须是可复用规则，不要写成流水账。

好例子：

```md
- 修改 Saber3 表单字段后，必须同步检查 `rules`、`search`、`dicData` 和 API 入参。
- BladeX 项目中，Controller 入参优先使用 DTO/VO，不直接暴露 Entity。
```

坏例子：

```md
- 今天修了登录问题。
- 用户让我改了 token。
```

### 7.9 Lessons 检查点

最终回复前必须执行 lessons 检查：

- 如果本次有用户纠正或长期规则，必须写入 `tasks/lessons.md`；
- 如果没有长期规则，最终回复中说明“本次无新增 lessons”。

不要沉默跳过。

### 7.10 状态文件最终报告

最终回复必须包含“状态文件”小节。

格式：

```md
## 状态文件

- `tasks/todo.md`：已创建 / 已更新 / 已清空 / 未使用，原因：...
- `tasks/archive/YYYY-MM.md`：已归档 / 未归档，原因：...
- `tasks/lessons.md`：已更新 / 无新增 lessons / 未使用，原因：...
```

## 8. Verification

任务完成前必须验证。

验证强度按任务类型选择。

| 任务类型 | 最低验证要求 |
|---|---|
| 纯文案 / 注释 / 配置说明 | 检查 diff |
| 后端小改动 | 运行相关单元测试，或至少编译相关模块 |
| 后端业务逻辑 | 单元测试 + 必要的集成验证 |
| 前端样式 / 字段小改动 | 构建或类型检查 |
| 前端交互逻辑 | 相关测试；必要时 Playwright e2e |
| 数据库结构变更 | SQL 变更脚本 + 兼容性说明 |
| 权限相关改动 | 验证有权限和无权限两类场景 |
| 状态流转改动 | 验证正常流转和非法状态流转 |

禁止使用以下表达代替验证：

- “应该没问题”
- “看起来可以”
- “理论上可行”
- “改动很小，不需要测”

如果无法运行测试，必须说明：

1. 为什么无法运行；
2. 已完成哪些静态检查；
3. 建议用户执行什么命令验证。

示例：

```text
未能运行测试，因为当前环境缺少 Maven。
建议在项目根目录执行：

mvn test -pl blade-auth
```

## 9. Bug Fixing

用户提供 bug、日志、异常栈、失败测试或错误现象时，不要等待用户手把手指导。

处理流程：

1. 阅读报错和相关日志；
2. 写入或更新 `tasks/todo.md`；
3. 定位相关代码；
4. 找到根因；
5. 最小范围修复；
6. 运行验证；
7. 归档任务结果；
8. 检查是否需要写入 `tasks/lessons.md`；
9. 总结结果。

修复完成后必须说明：

- 根因
- 修改点
- 涉及文件
- 验证方式
- 状态文件处理结果
- 是否存在残留风险

不要只修表面现象。

不要用临时兜底逻辑掩盖真实问题。

## 10. Database Rules

涉及数据库时必须谨慎。

### 10.1 表结构变更

任何表结构变更都必须提供 SQL。

包括：

- 新增表
- 修改表
- 新增字段
- 修改字段类型
- 修改索引
- 修改唯一约束
- 修改默认值
- 修改枚举字典

SQL 应包含：

- 正向变更 SQL
- 必要时提供回滚 SQL
- 默认值说明
- 兼容历史数据说明

### 10.2 字段设计

字段设计应考虑：

- 是否允许为空
- 默认值
- 索引
- 唯一性
- 字段长度
- 字典值
- 历史数据兼容
- 是否涉及租户隔离
- 是否涉及逻辑删除

### 10.3 MySQL 注意事项

避免使用 MySQL 关键字作为字段名。

例如：

- `desc`
- `index`
- `order`
- `group`
- `rank`
- `key`
- `value`

如果已有字段无法避免，SQL 中必须使用反引号。

## 11. Backend Rules

后端开发应遵守现有分层。

通用分层原则：

- Controller：处理请求、参数校验、权限入口、返回结果。
- Service：处理业务逻辑、事务、状态流转。
- Mapper / DAO：只处理数据访问。
- Entity：对应数据库表结构。
- DTO：用于接口入参或服务间传输。
- VO：用于接口返回。
- Query / Param：用于查询条件。

不要在 Controller 中堆业务逻辑。

不要在 Mapper 中写业务判断。

不要直接把 Entity 暴露给前端，除非项目既有风格就是如此。

新增接口时必须考虑：

- 权限控制
- 参数校验
- 异常处理
- 返回格式
- 日志记录
- 租户隔离
- 数据权限
- 幂等性
- 分页限制

涉及事务时：

- 明确事务边界；
- 不要把远程调用、耗时 IO 放进大事务；
- 注意异常回滚规则。

## 12. Frontend Rules

前端开发应保持现有项目风格。

修改页面时需要检查：

- 路由配置
- 菜单配置
- 权限编码
- API 调用
- 表单字段
- 校验规则
- 字典配置
- 列表搜索项
- 表格列
- 新增/编辑/详情页面一致性

不要随意引入新的 UI 库。

不要破坏现有布局风格。

表单字段修改时，通常需要同步检查：

- `form`
- `rules`
- `option`
- `column`
- `search`
- `dicData`
- `props`
- API 入参
- API 返回值

涉及交互逻辑时，需要说明验证方式。

---

## 13. Security Rules

涉及安全、认证、授权、数据权限时必须谨慎。

必须检查：

- 当前用户身份
- 角色权限
- 菜单权限
- 按钮权限
- 数据范围
- 租户隔离
- 越权访问
- 参数篡改
- 敏感字段返回
- 日志中是否泄露敏感信息

禁止：

- 为了调试硬编码 token、密码、密钥；
- 在日志中输出密码、token、密钥；
- 绕过权限校验；
- 使用前端控制代替后端权限控制。

## 验证结果

- 已运行：`命令`

## 状态文件

```md

```

## 15. Pre-Final Checklist

最终回复前必须检查：

- 是否修改了仓库文件？
- 如果修改了，是否写入过 `tasks/todo.md`？
- 如果任务完成，是否已清空 `tasks/todo.md`？
- 如果任务完成，是否已归档到 `tasks/archive/YYYY-MM.md`？
- 本次是否触发 `tasks/lessons.md` 规则？
- 如果触发，是否已更新 `tasks/lessons.md`？
- 是否运行了合适的验证？
- 如果无法验证，是否说明原因和建议命令？

任何一项未完成，都不要声称任务完成。

## 16. Hard Rules

以下规则必须始终遵守：

- 不要擅自创建说明文档。
- 状态文件不属于说明文档；符合规则时必须创建或更新。
- 对任何会修改仓库文件的任务，必须写入 `tasks/todo.md`。
- `tasks/todo.md` 只保存当前任务，任务完成后必须清空。
- 不要让 `tasks/todo.md` 长期累积历史任务。
- 任务完成后，如修改过仓库文件，必须归档到 `tasks/archive/YYYY-MM.md`。
- 用户纠正行为或提出长期偏好时，必须更新 `tasks/lessons.md`。
- 最终回复必须说明状态文件处理结果。
- 复杂任务确认前不要动业务代码。
- 修改代码必须基于真实代码依据。
- 未验证的任务不得声称完成。
- 不要用“应该没问题”代替验证。
- 不做无关重构。
- 不覆盖用户已有改动。
- 数据库结构变更必须提供 SQL。
